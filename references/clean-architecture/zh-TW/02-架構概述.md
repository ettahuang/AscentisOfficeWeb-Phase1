# 架構概述

> 本文說明 PaymentService 專案的架構遷移背景、舊架構問題分析，以及新架構（Clean Architecture）的設計理念與實作方式。

---

## 專案架構遷移狀態

### 遷移背景

PaymentService 專案正從傳統的三層式架構（Controller → BLL → DAL）遷移至 Clean Architecture。此次重構以 `GetDepositOptionsAsync` API 為標竿案例，建立團隊可依循的架構規範。

### 遷移範圍

| 階段 | 狀態 | 說明 |
|------|------|------|
| Deposit 相關 API | ✅ 已完成 | `GetDepositOptionsAsync` 完整重構 |
| Customer 相關查詢 | ✅ 已完成 | `CustomerRepository`、`CustomerContext` |
| Withdrawal 相關 API | 🔄 進行中 | 依循 Deposit 模式進行 |
| 其他舊有 API | ⏳ 待規劃 | 維持現狀，逐步遷移 |

### 新舊架構共存策略

```
PaymentService/
├── Service.BLL/          ← 舊架構 BLL（維持現狀）
├── Service.DAL/          ← 舊架構 DAL（維持現狀）
├── Models/Domain/        ← 新架構 Domain Layer
├── Services/             ← 新架構 Application Layer
└── Repositories/         ← 新架構 Infrastructure Layer
```

## 舊架構問題分析

### 問題一：Controller 職責過重

**現象**：Controller 包含業務邏輯、資料轉換、錯誤處理等多重職責。

```csharp
// ❌ 舊架構：Controller 做太多事
public class DepositController
{
    public async Task<IActionResult> GetDepositOptions(int custId)
    {
        // 1. 查詢客戶資料
        var cust = await _custService.GetCust(custId);

        // 2. 業務邏輯判斷（不應在 Controller）
        if (cust.VipLevel < 3 && cust.PointLevel < 100)
        {
            return BadRequest("資格不符");
        }

        // 3. 查詢存款通道
        var channels = await _depositService.GetChannels();

        // 4. 複雜的篩選邏輯（不應在 Controller）
        var filtered = channels.Where(c =>
            c.VipLevels.Contains(cust.VipLevel) &&
            !c.ExcludeTags.Intersect(cust.Tags).Any() &&
            c.MinAmount <= limit.MaxAmount
        ).ToList();

        return Ok(filtered);
    }
}
```

**問題**：
- Controller 超過 200 行
- 難以撰寫單元測試
- 業務邏輯散落各處，難以維護

### 問題二：資料模型混亂

**現象**：同一個類別同時作為資料庫映射、業務邏輯載體、API 回應格式。

```csharp
// ❌ 舊架構：一個類別多重用途
public class PayChannel
{
    // 資料庫欄位
    public int SysId { get; set; }
    public string VipLevel { get; set; } // JSON 字串

    // 業務邏輯方法
    public bool IsAvailable(Customer cust) { ... }

    // API 序列化屬性
    [JsonIgnore]
    public string InternalField { get; set; }
}
```

**問題**：
- 資料庫變更影響 API 回應
- 業務邏輯測試需 mock 資料庫
- 難以理解類別的真正職責

### 問題三：缺乏依賴反轉

**現象**：高層模組直接依賴低層實作。

```csharp
// ❌ 舊架構：直接依賴實作
public class PayBLL
{
    private readonly PayService _payService; // 直接依賴具體類別

    public PayBLL(PayService payService)
    {
        _payService = payService;
    }
}
```

**問題**：
- 無法替換實作（如 Mock）
- 單元測試困難
- 高耦合，修改一處影響多處

### 問題四：橫切關注點散落

**現象**：快取、日誌、異常處理邏輯散落在各層。

```csharp
// ❌ 舊架構：快取邏輯混入業務程式碼
public class PayService
{
    public async Task<List<Channel>> GetChannels(int siteId)
    {
        var cacheKey = $"Channels:{siteId}";
        var cached = await _redis.GetAsync<List<Channel>>(cacheKey);
        if (cached != null) return cached;

        var data = await _db.Query("...");
        await _redis.SetAsync(cacheKey, data, TimeSpan.FromMinutes(5));
        return data;
    }
}
```

**問題**：
- 每個方法都需處理快取
- 快取策略不一致
- 難以全面調整快取行為

## 新架構設計（Clean Architecture）

### 同心圓架構概念

```
        ┌─────────────────────────────────────┐
        │         Infrastructure              │
        │  ┌─────────────────────────────┐    │
        │  │       Application           │    │
        │  │  ┌─────────────────────┐    │    │
        │  │  │      Domain         │    │    │
        │  │  │  (核心業務邏輯)      │    │    │
        │  │  └─────────────────────┘    │    │
        │  │   Services, Use Cases       │    │
        │  └─────────────────────────────┘    │
        │   Repositories, External APIs       │
        └─────────────────────────────────────┘
                        ↑
              依賴方向：由外向內
```

### 各層對應專案結構

| 層級 | 專案路徑 | 職責 |
|------|----------|------|
| **Domain** | `Models/Domain/` | 核心業務邏輯、實體、值物件 |
| **Application** | `Services/` | Use Case、介面定義、協調邏輯 |
| **Infrastructure** | `Repositories/` | 資料存取、外部服務整合 |
| **Presentation** | `Controllers/` | HTTP 請求處理、回應格式化 |

### 依賴方向

```
Controller → IDepositQueryService → ICustomerRepository
                                 → IDepositRepository
                                         ↓
                               CustomerRepository (實作)
                               DepositRepository (實作)
```

**關鍵點**：
- Controller 依賴 Application Layer 的介面
- Application Layer 定義 Repository 介面
- Infrastructure Layer 實作這些介面
- **依賴方向由外向內，內層不知道外層存在**

## 新架構實作範例

### Domain Layer

```csharp
// Models/Domain/DepositChannel.cs
public class DepositChannel
{
    public CurrencyEnum Currency { get; init; }
    public DepositTypeEnum DepositType { get; init; }
    public List<int> VipLevels { get; init; }
    public decimal MinAmount { get; init; }
    public decimal MaxAmount { get; init; }

    // 業務邏輯封裝於 Domain Model
    public bool IsSupported(CustomerContext custContext)
    {
        return custContext.IsCurrencyOf(Currency)
               && custContext.IsReachedPointLevel(PointLevel)
               && custContext.IsVipLevelInScope(VipLevels)
               && !custContext.IsTagInScope(ExcludeTags);
    }

    // 不可變性：回傳新實例
    public DepositChannel WithAmountLimit(DepositLimitInfo limitInfo)
    {
        return new DepositChannel(
            Currency, DepositType, ExcludeTags, VipLevels, PointLevel,
            Math.Max(MinAmount, limitInfo.MinAmount),
            Math.Min(MaxAmount, limitInfo.MaxAmount),
            Sort);
    }
}
```

### Application Layer

```csharp
// Services/ICustomerRepository.cs（介面定義）
public interface ICustomerRepository
{
    Task<Customer> GetCustInfoAsync(int custId);
    Task<CustomerContext> GetCustContextAsync(int custId);
}

// Services/DepositQueryService.cs（Use Case 實作）
public class DepositQueryService : IDepositQueryService
{
    private readonly ICustomerRepository _customerRepository;
    private readonly IDepositRepository _depositRepository;

    public async Task<DepositChannelsByDepositType> GetDepositChannelsByDepositTypeAsync(int custId)
    {
        // 協調多個 Repository
        var custContext = await _customerRepository.GetCustContextAsync(custId);
        var depositChannels = await _depositRepository.GetDepositChannelsAsync(custContext.Customer.SiteId);

        // 委託業務邏輯給 Domain Model
        return depositChannels.GroupByDepositType(custContext);
    }
}
```

### Infrastructure Layer

```csharp
// Repositories/CustomerRepository.cs
public class CustomerRepository : ICustomerRepository
{
    private readonly MySqlClient _mainDbClient;

    public async Task<Customer> GetCustInfoAsync(int custId)
    {
        var dbCust = await _mainDbClient.QueryFirstAsync<DbModel.Customer>(
            "Customer_Get", new { CustId = custId });

        // 透過 CreateDomain() 轉換
        return dbCust?.CreateDomain();
    }
}

// Repositories/CustomerRepositoryDecorator.cs（快取層）
public class CustomerRepositoryDecorator : ICustomerRepository
{
    private readonly ICustomerRepository _customerRepository;
    private readonly ICacheHelper _cacheHelper;

    public async Task<Customer> GetCustInfoAsync(int custId)
    {
        return await _cacheHelper.TryGetOrCreateAsync(
            $"CustInfo:{custId}",
            TimeSpan.FromMinutes(1),
            async () => await _customerRepository.GetCustInfoAsync(custId)
        );
    }
}
```

### DI 註冊（Decorator 模式）

```csharp
// Startup.cs
services.AddScoped<ICustomerRepository, CustomerRepository>();
services.Decorate<ICustomerRepository, CustomerRepositoryDecorator>();

services.AddScoped<IDepositRepository, DepositRepository>();
services.Decorate<IDepositRepository, DepositRepositoryDecorator>();
```

## 橫切關注點設計

### Middleware（進入點處理）

處理所有進入系統的 HTTP 請求：

```
HTTP Request → [GlobalExceptionMiddleware] → [RequestLogMiddleware] → Controller
                        ↓
               統一異常處理、日誌記錄
```

### DelegatingHandler（輸出點處理）

處理所有對外的 HTTP 呼叫：

```
HttpClient → [LoggingHandler] → [RetryHandler] → [TimeoutHandler] → External API
                  ↓
        請求日誌、重試機制、超時控制
```

## 新舊架構對比總結

| 面向 | 舊架構 | 新架構 |
|------|--------|--------|
| **業務邏輯位置** | 散落在 Controller、BLL | 集中於 Domain Model |
| **資料模型** | 單一類別多用途 | Domain/DbModel/DTO 分離 |
| **依賴方向** | 高層依賴低層實作 | 依賴反轉，依賴介面 |
| **快取處理** | 混入業務程式碼 | Decorator 模式分離 |
| **測試難度** | 高（需 Mock 資料庫） | 低（可 Mock 介面） |
| **可維護性** | 修改一處影響多處 | 各層獨立，影響最小化 |

## 遷移策略

### 漸進式遷移原則

1. **新功能**：直接使用新架構
2. **Bug 修復**：評估是否順便重構
3. **重大變更**：規劃完整重構

### 遷移檢查清單

- [ ] 識別現有程式碼的 Domain Model
- [ ] 將 DbModel 與 Domain Model 分離
- [ ] 定義 Repository 介面於 Application Layer
- [ ] 實作 Repository 於 Infrastructure Layer
- [ ] 使用 Decorator 模式處理快取
- [ ] 撰寫單元測試驗證業務邏輯
- [ ] 更新 DI 註冊

## ✅ 新人常見踩雷點

### 1. 將新程式碼放錯位置

```csharp
// ❌ 錯誤：在舊架構目錄新增 Clean Architecture 程式碼
Service.DAL/NewCustomerRepository.cs  // 不對！

// ✅ 正確：使用新架構目錄
Repositories/CustomerRepository.cs
```

### 2. 混用新舊架構

```csharp
// ❌ 錯誤：新架構 Service 直接依賴舊架構 DAL
public class DepositQueryService
{
    private readonly PayService _payService; // 舊架構
}

// ✅ 正確：透過介面抽象
public class DepositQueryService
{
    private readonly IDepositRepository _depositRepository; // 新架構介面
}
```

### 3. 忽略 CreateDomain 轉換

```csharp
// ❌ 錯誤：直接回傳 DbModel
public async Task<DbModel.Customer> GetCustInfoAsync(int custId)
{
    return await _db.QueryFirstAsync<DbModel.Customer>(...);
}

// ✅ 正確：轉換為 Domain Model
public async Task<Customer> GetCustInfoAsync(int custId)
{
    var dbModel = await _db.QueryFirstAsync<DbModel.Customer>(...);
    return dbModel?.CreateDomain();
}
```
