# TDD 與 Tidy First 實踐指南

> 本文基於 Kent Beck 的 Test-Driven Development (TDD) 與 Tidy First 原則，說明團隊的開發方法論。這是確保程式碼品質與可維護性的核心實踐。

---

## 核心開發原則

- 始終遵循 TDD 循環：Red → Green → Refactor
- 先撰寫最簡單的失敗測試
- 實作剛好讓測試通過的最少程式碼
- 測試通過後才進行重構
- 遵循 "Tidy First" 方法：結構性變更與行為變更分離
- 全程維持高品質程式碼

---

## TDD 方法論

### Red → Green → Refactor 循環

```
Red (紅燈)
    │ 撰寫一個會失敗的測試
    ▼
Green (綠燈)
    │ 撰寫剛好讓測試通過的程式碼
    ▼
Refactor (重構)
    │ 改善程式碼結構，保持測試通過
    ▼
重複循環
```

### TDD 實踐要點

| 原則 | 說明 |
|------|------|
| **一次一個測試** | 從定義小功能增量的失敗測試開始 |
| **有意義的測試名稱** | 描述行為，如 `ShouldSumTwoPositiveNumbers` |
| **清晰的失敗訊息** | 讓測試失敗時能清楚知道問題所在 |
| **最小實作** | 只撰寫讓測試通過的程式碼，不多不少 |
| **測試通過後再重構** | 確保有安全網後才改善結構 |

### 修復缺陷的流程

```
發現 Bug
    │
    ├─ 1. 先撰寫一個 API 層級的失敗測試
    │      └─ 確認 Bug 可被測試捕捉
    │
    ├─ 2. 再撰寫最小範圍的單元測試來重現問題
    │      └─ 精確定位問題所在
    │
    └─ 3. 修復 Bug，讓兩個測試都通過
```

## Tidy First 方法

### 變更類型分離

將所有變更分為兩種類型，**絕不混合**：

| 類型 | 說明 | 範例 |
|------|------|------|
| **結構性變更 (Structural)** | 重新排列程式碼，不改變行為 | 重新命名、抽取方法、搬移程式碼、調整格式 |
| **行為變更 (Behavioral)** | 新增或修改實際功能 | 新功能、Bug 修復、業務邏輯調整 |

### 為什麼要分離？

| 優勢 | 說明 |
|------|------|
| **易於 Code Review** | 審查者可以快速確認結構性變更沒有改變行為 |
| **易於追蹤** | Git 歷史清楚記錄每次變更的意圖 |
| **易於回滾** | 出問題時可以精確回滾特定變更 |
| **降低風險** | 分開處理降低引入 Bug 的風險 |

### Tidy First 工作流程

```
需要同時進行結構性與行為變更時：
    │
    ├─ 1. 先進行結構性變更
    │      └─ 執行測試驗證行為未改變
    │
    ├─ 2. 提交結構性變更
    │      └─ commit message: "refactor: ..."
    │
    ├─ 3. 再進行行為變更
    │      └─ 執行測試驗證新行為
    │
    └─ 4. 提交行為變更
           └─ commit message: "feat: ..." 或 "fix: ..."
```

### 實際範例

**情境**：需要在 `DepositService` 中新增一個功能，但發現現有程式碼需要先重構。

```
❌ 錯誤做法：一次提交混合結構性與行為變更

git commit -m "feat: add deposit validation and refactor service"
```

```
✅ 正確做法：分開提交

# 第一步：結構性變更
git commit -m "refactor: extract validation logic to separate method"

# 第二步：行為變更
git commit -m "feat: add deposit amount validation"
```

## Commit 紀律

### 提交條件

只在以下條件都滿足時才提交：

- [ ] 所有測試都通過
- [ ] 所有編譯器/Linter 警告都已解決
- [ ] 變更代表單一邏輯工作單元
- [ ] 提交訊息明確說明是結構性或行為變更

### 提交原則

| 原則 | 說明 |
|------|------|
| **小而頻繁** | 優先於大而少的提交 |
| **單一職責** | 一個提交只做一件事 |
| **類型標示** | 明確標示變更類型 |
| **可獨立運行** | 每個提交後專案都應該可以正常運行 |

### Commit Message 與變更類型對應

| 變更類型 | Commit Prefix | 範例 |
|---------|---------------|------|
| 結構性變更 | `refactor:` | `refactor: rename GetCustInfo to GetCustomerById` |
| 結構性變更 | `style:` | `style: format code according to guidelines` |
| 行為變更 | `feat:` | `feat: add VIP level validation` |
| 行為變更 | `fix:` | `fix: correct deposit limit calculation` |

## 程式碼品質標準

| 標準 | 說明 | 實踐方式 |
|------|------|---------|
| **消除重複** | 徹底消除重複程式碼 | 抽取共用方法、使用繼承或組合 |
| **表達意圖** | 透過命名和結構清楚表達意圖 | 使用領域語言命名 |
| **明確依賴** | 讓依賴關係顯而易見 | 透過建構子注入依賴 |
| **小而專注** | 方法保持小巧，專注單一職責 | 一個方法只做一件事 |
| **最小狀態** | 最小化狀態和副作用 | 優先使用不可變物件 |
| **最簡方案** | 使用可能有效的最簡單方案 | 避免過度設計 |

## 重構指南

### 重構時機

| 時機 | 說明 |
|------|------|
| ✅ **Green 階段** | 測試通過後才進行重構 |
| ✅ **發現重複** | 看到重複程式碼時 |
| ✅ **命名不清** | 發現命名無法表達意圖時 |
| ❌ **測試失敗時** | 先讓測試通過，再重構 |
| ❌ **趕時間時** | 重構需要時間和專注力 |

### 重構原則

| 原則 | 說明 |
|------|------|
| **使用正確名稱** | 使用已建立的重構模式及其正確名稱（如 Extract Method） |
| **一次一個** | 一次只做一個重構變更 |
| **頻繁測試** | 每個重構步驟後執行測試 |
| **優先順序** | 優先移除重複或提升清晰度的重構 |

### 常見重構模式

| 模式 | 說明 | 使用時機 |
|------|------|---------|
| **Extract Method** | 抽取方法 | 方法過長或有重複邏輯 |
| **Rename** | 重新命名 | 名稱無法表達意圖 |
| **Move Method** | 搬移方法 | 方法放在錯誤的類別 |
| **Extract Class** | 抽取類別 | 類別職責過多 |
| **Inline** | 內聯 | 抽象層級不必要 |

## 工作流程範例

### 實作新功能

```
1. 為功能的一小部分撰寫簡單的失敗測試
    ↓
2. 實作剛好讓測試通過的最小程式碼
    ↓
3. 執行測試確認通過 (Green)
    ↓
4. 評估是否需要重構
   ├─ 需要 → 進行結構性變更，執行測試，提交
   └─ 不需要 → 繼續下一步
    ↓
5. 提交行為變更
    ↓
6. 為下一個小功能增量新增測試
    ↓
7. 重複直到功能完成
```

### 修復 Bug

```
1. 撰寫重現 Bug 的測試（確認測試失敗）
    ↓
2. 修復 Bug（讓測試通過）
    ↓
3. 評估是否需要重構以防止類似 Bug
   ├─ 需要 → 分開提交：先 fix，再 refactor
   └─ 不需要 → 直接提交 fix
    ↓
4. 提交變更
```

## 與現有規範的整合

| 規範 | 與 TDD/Tidy First 的關係 |
|------|--------------------------|
| **Git Commit Message 規範** | 使用 `refactor:` 標示結構性變更，`feat:`/`fix:` 標示行為變更 |
| **Code Review 檢查清單** | 確認結構性與行為變更是否分離 |
| **測試規範與指南** | TDD 是測試的核心方法論 |
| **CSharp 程式碼編寫規範** | 重構時遵循編碼規範 |

## 常見問題

### Q: 小型變更也需要分離結構性與行為變更嗎？

**A**: 是的，即使是小型變更也建議分離。這個習慣會讓 Git 歷史更清晰，長期來看會節省更多時間。

### Q: 如果重構過程中發現需要改變行為怎麼辦？

**A**: 停下來，先提交目前的結構性變更，然後開始新的行為變更。不要在同一個提交中混合。

### Q: TDD 會不會讓開發變慢？

**A**: 短期可能會稍慢，但長期會更快。TDD 減少了 Debug 時間、降低了 Bug 率、讓重構更有信心。

## ✅ Review Checklist

### TDD 檢查

- [ ] 是否先撰寫失敗測試再實作功能？
- [ ] 測試名稱是否清楚描述行為？
- [ ] 是否只實作剛好讓測試通過的程式碼？

### Tidy First 檢查

- [ ] 結構性變更與行為變更是否分開提交？
- [ ] 每個提交是否都能獨立運行且測試通過？
- [ ] Commit message 是否正確標示變更類型？

### 程式碼品質檢查

- [ ] 是否消除了重複程式碼？
- [ ] 命名是否清楚表達意圖？
- [ ] 方法是否保持小巧專注？
