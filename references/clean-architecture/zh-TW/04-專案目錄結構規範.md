# 專案目錄結構規範

> 本文說明專案的目錄結構規範，包含各目錄的職責定義、命名規則，以及如何正確放置新增的類別。適用於 Clean Architecture 專案（如 PaymentService、BonusService）。

## 目錄深度規則

### 規則：目錄深度 ≤ 3 層

```
PaymentService/           # 第 1 層
├── Models/               # 第 2 層
│   ├── Domain/           # 第 3 層
│   │   └── Customer.cs   # 檔案（不算層級）
```

### 正確示範

```
✅ PaymentService/Models/Domain/Customer.cs          (3 層)
✅ PaymentService/Services/DepositQueryService.cs    (2 層)
✅ PaymentService/Repositories/CustomerRepository.cs (2 層)
```

### 錯誤示範

```
❌ PaymentService/Models/Domain/Customer/Entities/Customer.cs  (5 層)
❌ PaymentService/Features/Deposit/Handlers/V1/Handler.cs       (5 層)
```

### 例外情況

若確實需要更深層級（如 Feature 目錄），應重新評估結構或使用命名空間區分：

```
// 使用命名空間區分，而非深層目錄
namespace PaymentService.Features.Deposit.Handlers
{
    public class GetDepositOptionsV1Handler { }
    public class GetDepositOptionsV2Handler { }
}
```

### Models/ValueObject/

| 項目 | 說明 |
|------|------|
| **職責** | 值物件 - 無身份識別的不可變值類型 |
| **放置內容** | 以屬性值判斷相等的類別 |
| **命名規則** | `{Name}.cs`（名詞，描述其概念） |

> 詳細的 Value Object 設計指南請參考 [08-Domain-Model-設計指南](./08-Domain-Model-設計指南.md#值物件value-objects設計)

```csharp
// Models/ValueObject/Period.cs
namespace PaymentService.Models.ValueObject
{
    /// <summary>
    /// 時間區間值物件
    /// </summary>
    public record Period
    {
        public DateTime StartTime { get; init; }
        public DateTime EndTime { get; init; }

        public bool Contains(DateTime dateTime)
        {
            return dateTime >= StartTime && dateTime <= EndTime;
        }
    }
}

// Models/ValueObject/VisibilityRules.cs
namespace PaymentService.Models.ValueObject
{
    /// <summary>
    /// 可見性規則值物件
    /// </summary>
    public record VisibilityRules
    {
        public bool RequiresLogin { get; init; }
        public IReadOnlyList<int> AllowedVipLevels { get; init; }

        public bool IsVisibleTo(ViewerContext viewer) { ... }
    }
}
```

### Services/

| 項目 | 說明 |
|------|------|
| **職責** | Application Layer - 介面定義 + Service 實作 |
| **放置內容** | Repository 介面、Service 介面與實作 |
| **命名規則** | `I{Entity}Repository.cs`、`I{Name}Service.cs`、`{Name}Service.cs` |

#### 介面放置方式

有兩種組織方式，專案可根據規模選擇：

| 方式 | 適用情境 | 結構 |
|------|---------|------|
| **直接放 Services/** | 小型專案、介面數量少 | `Services/ICustomerRepository.cs` |
| **使用 Abstractions/ 子目錄** | 中大型專案、介面數量多 | `Services/Abstractions/ICustomerRepository.cs` |

```csharp
// 方式一：直接放 Services/（小型專案）
// Services/ICustomerRepository.cs
namespace PaymentService.Services
{
    public interface ICustomerRepository
    {
        Task<Customer> GetCustInfoAsync(int custId);
    }
}

// 方式二：使用 Abstractions/ 子目錄（中大型專案）
// Services/Abstractions/ICustomerRepository.cs
namespace PaymentService.Services.Abstractions
{
    public interface ICustomerRepository
    {
        Task<Customer> GetCustInfoAsync(int custId);
    }
}

// Service 實作（無論哪種方式都放在 Services/）
// Services/DepositQueryService.cs
namespace PaymentService.Services
{
    public class DepositQueryService : IDepositQueryService
    {
        // ...
    }
}
```

### ApiClients/Payload/

| 項目 | 說明 |
|------|------|
| **職責** | ApiClient 專用的 Request/Response 模型（與外部系統通訊的 DTO） |
| **放置內容** | ApiClient 呼叫外部 API 時使用的請求與回應類別 |
| **命名規則** | `{ServiceName}Request.cs`、`{ServiceName}Response.cs` |

> **與 Models/Payload/ 的區別**：`Models/Payload/` 存放本服務對外公開的 API 模型；`ApiClients/Payload/` 存放呼叫外部服務時使用的模型。

```csharp
        public decimal Amount { get; set; }

}

// ApiClients/Payload/PaymentGatewayResponse.cs
namespace PaymentService.ApiClients.Payload
{
    public class PaymentGatewayResponse
    {
        public string TransactionId { get; set; }
        public string Status { get; set; }
        public string RedirectUrl { get; set; }
    }
}
```

### Adapters/

| 項目 | 說明 |
|------|------|
| **職責** | 封裝第三方 SDK / DLL 的使用，提供統一介面 |
| **放置內容** | Adapter 類別 |
| **命名規則** | `{Name}Adapter.cs` |

```csharp
// Adapters/ExcelAdapter.cs
namespace PaymentService.Adapters
{
    public class ExcelAdapter : IExcelAdapter
    {
        public byte[] GenerateExcel<T>(IEnumerable<T> data, string sheetName)
        {
            // 封裝 EPPlus / NPOI SDK 操作
            using var package = new ExcelPackage();
            var worksheet = package.Workbook.Worksheets.Add(sheetName);
            // ...
            return package.GetAsByteArray();
        }

        public IEnumerable<T> ParseExcel<T>(Stream excelStream)
        {
            // 封裝 EPPlus / NPOI SDK 操作
        }
    }
}

// Adapters/QrCodeAdapter.cs
namespace PaymentService.Adapters
{
    public class QrCodeAdapter : IQrCodeAdapter
    {
        public byte[] GenerateQrCode(string content, int pixelsPerModule = 20)
        {
            // 封裝 QRCoder SDK 操作
            using var qrGenerator = new QRCodeGenerator();
            var qrCodeData = qrGenerator.CreateQrCode(content, QRCodeGenerator.ECCLevel.Q);
            using var qrCode = new PngByteQRCode(qrCodeData);
            return qrCode.GetGraphic(pixelsPerModule);
        }
    }
}
```

## 命名規範

### 檔案命名

| 類型 | 命名規則 | 範例 |
|------|----------|------|
| Domain Model | `{Entity}.cs` | `Customer.cs` |
| Domain Model 集合 | `{Entity}s.cs` 或 `{Entity}List.cs` | `DepositChannels.cs`、`PromotionList.cs` |
| Value Object | `{Name}.cs` | `Period.cs`、`VisibilityRules.cs` |
| DbModel | `{Entity}.cs` | `Customer.cs`（不同命名空間） |
| Enum | `{Name}Enum.cs` | `CurrencyEnum.cs` |
| Repository 介面 | `I{Entity}Repository.cs` | `ICustomerRepository.cs` |
| Repository 實作 | `{Entity}Repository.cs` | `CustomerRepository.cs` |
| Decorator | `{Entity}RepositoryDecorator.cs` | `CustomerRepositoryDecorator.cs` |
| ApiClient 介面 | `I{ServiceName}ApiClient.cs` | `IPaymentGatewayApiClient.cs` |
| ApiClient 實作 | `{ServiceName}ApiClient.cs` | `PaymentGatewayApiClient.cs` |
| ApiClient Payload | `{ServiceName}Request.cs`、`{ServiceName}Response.cs` | `PaymentGatewayRequest.cs` |
| Adapter 介面 | `I{Name}Adapter.cs` | `IExcelAdapter.cs` |
| Adapter 實作 | `{Name}Adapter.cs` | `ExcelAdapter.cs` |
| Service 介面 | `I{Name}Service.cs` | `IDepositQueryService.cs` |
| Service 實作 | `{Name}Service.cs` | `DepositQueryService.cs` |
| Controller | `{Entity}Controller.cs` | `CustomerController.cs` |
| Middleware | `{Name}Middleware.cs` | `GlobalExceptionMiddleware.cs` |
| Exception | `{Name}Exception.cs` | `CustomerNotFoundException.cs` |
| Extension | `{Type}Extensions.cs` | `StringExtensions.cs` |

### Enum 後綴規則

所有列舉類別必須加上 `Enum` 後綴：

```csharp
// ✅ 正確：有 Enum 後綴
public enum CurrencyEnum { }
public enum DepositTypeEnum { }
public enum ErrorSeverityEnum { }

// ❌ 錯誤：缺少 Enum 後綴
public enum Currency { }
public enum DepositType { }
```

## 新增類別的決策流程

### 決策流程圖

```
新增類別
    │
    ├─ 是具有身份識別的業務實體嗎？
    │   ├─ 是 → Models/Domain/
    │   └─ 否 ↓
    │
    ├─ 是無身份識別的值類型嗎？（如時間區間、規則集合）
    │   ├─ 是 → Models/ValueObject/
    │   └─ 否 ↓
    │
    ├─ 是資料庫映射嗎？
    │   ├─ 是 → Models/DbModel/
    │   └─ 否 ↓
    │
    ├─ 是 API Request/Response 嗎？
    │   ├─ 是 → Models/Payload/
    │   └─ 否 ↓
    │
    ├─ 是列舉嗎？
    │   ├─ 是 → Models/Enum/
    │   └─ 否 ↓
    │
    ├─ 是 Repository 介面嗎？
    │   ├─ 是 → Services/ 或 Services/Abstractions/
    │   └─ 否 ↓
    │
    ├─ 是 Repository 實作嗎？
    │   ├─ 是 → Repositories/
    │   └─ 否 ↓
    │
    ├─ 是外部系統通訊客戶端嗎？（HTTP、Socket、gRPC）
    │   ├─ 是 → ApiClients/
    │   └─ 否 ↓
    │
    ├─ 是呼叫外部 API 的 Request/Response 模型嗎？
    │   ├─ 是 → ApiClients/Payload/
    │   └─ 否 ↓
    │
    ├─ 是第三方 SDK / DLL 的封裝嗎？
    │   ├─ 是 → Adapters/
    │   └─ 否 ↓
    │
    ├─ 是 Service 嗎？
    │   ├─ 是 → Services/
    │   └─ 否 ↓
    │
    ├─ 是 Controller 嗎？
    │   ├─ 是 → Controllers/
    │   └─ 否 ↓
    │
    ├─ 是 Middleware 嗎？
    │   ├─ 是 → Middlewares/
    │   └─ 否 ↓
    │
    ├─ 是 Exception 嗎？
    │   ├─ 是 → Exceptions/
    │   └─ 否 ↓
    │
    └─ 是 Extension 嗎？
        ├─ 是 → Extensions/
        └─ 否 → 評估是否需要新目錄
```

| ApiClient 專用 Payload | `ApiClients/Payload/` | `{ServiceName}Request.cs` |

| Controller | `Controllers/` | `{Entity}Controller.cs` |
| 自定義例外 | `Exceptions/` | `{Name}Exception.cs` |

## ✅ Review Checklist

### 目錄結構檢查

- [ ] 新類別是否放在正確的目錄？
- [ ] 目錄深度是否 ≤ 3 層？
- [ ] 命名是否符合規範？
- [ ] Enum 是否有 `Enum` 後綴？

### 命名空間檢查

- [ ] 命名空間是否與目錄結構一致？
- [ ] Repository 介面是否在 `Services`（或 `Services.Abstractions`）命名空間？
- [ ] Repository 實作是否在 `Repositories` 命名空間？

### 職責檢查

- [ ] Domain Model（有 ID）是否在 `Models/Domain/`？
- [ ] Value Object（無 ID）是否在 `Models/ValueObject/`？
- [ ] DbModel 是否在 `Models/DbModel/`？

## ✅ 新人常見踩雷點

### 1. Repository 介面放在 Repositories 目錄

```
❌ 錯誤：
Repositories/
├── ICustomerRepository.cs    # 介面不應該在這裡
└── CustomerRepository.cs

✅ 正確：
Services/
└── ICustomerRepository.cs    # 介面在 Services
Repositories/
└── CustomerRepository.cs     # 實作在 Repositories
```

### 2. 忘記 Enum 後綴

```csharp
// ❌ 錯誤：缺少 Enum 後綴
public enum Currency { CNY, USD }

// ✅ 正確：加上 Enum 後綴
public enum CurrencyEnum { CNY, USD }
```

### 3. 目錄層級過深

```
❌ 錯誤：5 層目錄
PaymentService/Models/Domain/Customer/Entities/Customer.cs

✅ 正確：3 層目錄
PaymentService/Models/Domain/Customer.cs
```

### 4. 在舊架構目錄新增 Clean Architecture 類別

```
❌ 錯誤：在舊架構目錄新增
Service.DAL/CustomerRepository.cs

✅ 正確：在新架構目錄新增
Repositories/CustomerRepository.cs
```

### 5. DbModel 與 Domain Model 混用

```csharp
// ❌ 錯誤：Domain Model 放在 DbModel 目錄
namespace PaymentService.Models.DbModel
{
    public class Customer  // 這是 Domain Model！
    {
        public bool IsVipLevelInScope(...) { ... }  // 業務方法
    }
}

// ✅ 正確：分開放置
namespace PaymentService.Models.DbModel
{
    public class Customer
    {
        public int VipLevel { get; init; }
        public Domain.Customer CreateDomain() { ... }
    }
}

namespace PaymentService.Models.Domain
{
    public class Customer
    {
        public int VipLevel { get; init; }
        public bool IsVipLevelInScope(...) { ... }
    }
}
```

## ✅ TL / Reviewer 檢查重點

### 結構審查

| 檢查項目 | 警示信號 |
|----------|----------|
| 目錄深度 | 超過 3 層的目錄結構 |
| 介面位置 | `IXxxRepository.cs` 在 `Repositories/` 目錄 |
| 舊架構混用 | 新類別放在 `Service.DAL/` 或 `Service.BLL/` |
| 命名不一致 | Enum 沒有 `Enum` 後綴 |

### 命名空間審查

- [ ] 類別的命名空間是否與檔案路徑一致？
- [ ] 是否有跨層的 using（如 Domain 引用 Repositories）？

### 職責邊界審查

- [ ] `Models/Domain/` 中的類別是否只包含業務邏輯？
- [ ] `Models/DbModel/` 中的類別是否只有資料映射？
- [ ] `Services/` 中是否只有介面和 Service 實作？
